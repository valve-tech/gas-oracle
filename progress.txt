## Key architectural decisions this session

- **chain-source is fully self-contained at this stage.** Has its
  own copies of `safeRequest`, `BlockResult`, `RawTx`, etc. — types
  are structurally identical to the gas-oracle ones so step 2's
  migration is a simple import swap, not a type-shape change.
  Splitting this way keeps each PR reviewable and avoids breaking
  gas-oracle while chain-source is half-built.

- **Capability probe is structural, not live.** Inspects
  `typeof transport.subscribe === 'function'` rather than issuing a
  real `eth_subscribe`. Trade-off: a transport that exposes
  `subscribe()` but whose upstream rejects the channel will report
  `'subscription'` at construction and discover the truth lazily.
  Mid-session capability degradation surfaces via `signal-degraded`
  events on downstream consumers (tx-tracker) — chain-source itself
  emits no editorial events.

- **Eager probe at construction (not at start).** The probe runs
  immediately on `createChainSource()`, fire-and-forget. By the time
  any consumer reaches `start()` and the first tick fires, the probe
  has typically landed. Brief window where `capabilities()` returns
  conservative defaults; `await source.ready()` guarantees real values.

- **Two methods added beyond spec:** `pollOnce()` (out-of-band cycle
  driver — useful for serverless flows and deterministic test setup
  without fake timers) and `ready()` (awaits the eager probe). Both
  additive; the spec'd interface is fully implemented.

- **WS push (`eth_subscribe`) deliberately deferred.** The capability
  matrix discloses structural support, but the source always uses
  its interval poll cycle in this revision. Future PR adds the push
  path without changing consumer-facing surface.

- **`subscribeMempool` does not fire on `null` txpool.** When the
  upstream gates `txpool_content` for a tick, mempool subscribers do
  not receive an "empty mempool" event — that would be misleading.
  Block subscribers still fire (they're independent paths).

- **Local `dist/` is gitignored** but rebuilds touch
  `tsconfig.tsbuildinfo` files in each package's root — also
  gitignored. Don't be alarmed when they show up after `yarn build`.

- **Vitest binary is at root `node_modules/.bin/vitest`** (hoisted),
  not in per-package `node_modules`. The package-level `yarn test`
  works because yarn workspaces resolve it; running vitest manually
  needs `node node_modules/.bin/vitest run --root packages/<pkg>`.

  is required to keep destructuring `[method]` typed in test
  assertions; the LSP flagged it once before I added the explicit
