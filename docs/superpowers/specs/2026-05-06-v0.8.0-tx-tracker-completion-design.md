# v0.8.0 — tx-tracker contract completion + WS-push transports

**Status:** Design proposal. Approved scope, picks ratified 2026-05-06.
**Target release:** `v0.8.0` across all 5 `@valve-tech/*` packages (synced).
**Shape:** single mega-PR, many small verified commits, one signed tag at the end.

---

## 1. Goal

Close the remaining gaps in `@valve-tech/tx-tracker`'s spec contract,
land the WS-push transports in `@valve-tech/chain-source` that the
capability-disclosure machinery already anticipates, and upstream the
two transaction-verbs from Provex that complete the package's surface.

After this release the toolkit has:

- **No type-vs-runtime gaps** in tx-tracker's `LostSignalPolicy` —
  `receipt-poll-fallback` actually runs.
- **All three `tx-tracker-spec.md` "deferred to v0.4.0+" items closed**
  — receipt enrichment (§18.2), cross-tx correlation (§18.1), with
  multi-chain (§18.3) explicitly remaining deferred.
- **Push-side transports** (`eth_subscribe('newHeads')`,
  `'newPendingTransactions'`) in `chain-source`, surfaced through the
  existing capability matrix and exercised by the existing
  capability-matrix tests in tx-tracker.
- **`watchTransaction` and `replaceTransaction`** as top-level exports
  from tx-tracker, the natural transaction-verbs pair alongside `track`.

### F1 — `receipt-poll-fallback` `lostSignalPolicy` strategy

Spec reference: `docs/tx-tracker-spec.md` §8.

The type already accepts:

```ts
{
  strategy: 'receipt-poll-fallback'
  pollEveryBlocks: number
}
```

Today the runtime falls back to `'emit-uncertain'`. This release wires
the actual behavior: on every `pollEveryBlocks`-th new block, while a
tracked subscription is in a degraded state, force a `getReceipt(hash)`
call against `source`. Emit `seen-in-block` (with `source: 'receipt-poll'`)
on hit. Capability gate: requires
`source.capabilities().receiptByHash === 'available'` — if unavailable,
fall back to `'emit-uncertain'` with a one-shot warning via `onError`
explaining the gate failure.

Per-subscription override via `TrackOptions.lostSignalPolicy` is already
wired and unchanged.

### F2 — `withReceipts: true` opt-in receipt enrichment

Spec reference: `docs/tx-tracker-spec.md` §18.2 (closes deferral).

**Shape: eager opt-in.** Decided after considering lazy-accessor and
both-modes alternatives.

```ts
interface TrackOptions {
  // ...existing fields...
  /**
   * When true, fetch the transaction receipt at `seen-in-block` time
   * and attach it to the event via the new `receipt` field. Adds one
   * RPC per inclusion. Default false.
   */
  withReceipts?: boolean
}

namespace TxEvent {
  interface SeenInBlock {
    // ...existing fields...
    /** Present iff the subscription set `withReceipts: true`. */
    receipt?: TransactionReceipt
  }
}
```

**Why eager over lazy:** predictable RPC budget, simpler types, no
bigint↔hex coercion at lazy-accessor boundaries. If a consumer doesn't
want receipts on every inclusion, they leave the option off and call
`source.getReceipt(hash)` themselves on the events they care about.

**Capability gate:** mirrors F1 — requires `receiptByHash === 'available'`.
If unavailable while `withReceipts: true` is set on a subscription, emit
a one-shot warning via `onError` and degrade to receipt-less events
(events still flow; `receipt` field stays absent).

**Reorg semantics:** if a `vanished-from-block` fires after a
`seen-in-block` that carried a receipt, no special invalidation needed —
the receipt was for the now-uncanonical block, and that's fine, the
event already fired. A subsequent re-inclusion (`seen-in-block` for the
same hash on a different block) issues a fresh `getReceipt` and carries
the new receipt. Receipts are not cached across reorgs; each
`seen-in-block` gets its own fetch.

### F3 — `tracker.group([...hashes])` cross-tx correlation

Spec reference: `docs/tx-tracker-spec.md` §18.1 (closes deferral).

**Shape: composite events.** Decided after considering pass-through-with-tag
and both-modes alternatives.

New API on `TxTracker`:

```ts
interface TxTracker {
  // ...existing methods...

  /**
   * Track a logical group of related hashes (e.g., a wallet's
   * "claim + swap" pair). Emits group-level synthesis events
   * derived from the per-member event streams.
   */
  group(hashes: Hash[], options?: GroupOptions): TxGroupSubscription
}

interface GroupOptions {
  /** Optional human-readable group ID echoed in events. Default: random. */
  groupId?: string
  /** Per-member TrackOptions applied to each hash. */
  memberOptions?: TrackOptions
}

interface TxGroupSubscription {
  /** Async iterator of group-level events. */
  events(): AsyncIterable<TxGroupEvent>
  /** Callback shape. Returns unsubscribe handle. */
  subscribe(cb: (event: TxGroupEvent) => void): () => void
  /** Snapshot — current per-member statuses keyed by hash. */
  snapshot(): Record<Hash, TxStatus | null>
  /** Tear down (also tears down per-member subscriptions). */
  stop(): void
}

type TxGroupEvent =
  | { kind: 'group-progress';  groupId: string; confirmed: number; total: number; lastHash: Hash; at: { blockNumber: bigint } }
  | { kind: 'group-complete';  groupId: string; total: number; at: { blockNumber: bigint } }
  | { kind: 'group-failed';    groupId: string; failedHash: Hash; reason: 'dropped' | 'failed' | 'replaced'; at: { blockNumber: bigint } }
  | { kind: 'group-stopped';   groupId: string }
```

**Semantics:**

- `group-progress` fires when any member transitions to confirmed
  (`seen-in-block` with confirmations ≥ 1) for the first time.
- `group-complete` fires when all N members are confirmed.
- `group-failed` fires on the first member to drop / fail / be replaced.
  Subsequent failures don't re-fire (the group is already in a terminal
  state).
- After `group-complete` or `group-failed`, the group subscription is
  effectively terminal — callers can call `stop()` to release resources
  but no further events fire.

**Why composite over pass-through:** pass-through is what consumers
already get from per-hash subscribe + their own dispatcher. The point
of `group()` is the synthesis.

**Replacement does not auto-promote.** If a member's hash is replaced
(via same-nonce bump), the group emits `group-failed` with
`reason: 'replaced'` — the original hash is terminally gone from the
group's perspective. The group does NOT automatically pick up the
replacement hash and continue tracking. If the consumer wants the
replacement to count, they construct a new `group()` with the new hash.
Rationale: auto-promotion silently changes which set of hashes the
group represents, and that's a `chainOfHashes` semantics question we'd
rather not answer implicitly.

### F4 — `watchTransaction` top-level export

Provex upstream item 4 (`upstream-candidates.md:43`).

```ts
import { watchTransaction } from '@valve-tech/tx-tracker'

interface WatchTransactionOptions {
  client: PublicClient
  hash: Hash
  /** Required confirmations before onMined fires. Default 1. */
  confirmations?: number
  /** Blocks of "no observation" before onDropped fires. Default 12. */
  staleAfterBlocks?: number
  onMined?: (event: TxEvent.SeenInBlock) => void
  onDropped?: () => void
  onError?: (method: string, err: unknown) => void
}

function watchTransaction(options: WatchTransactionOptions): () => void
```

Internally constructs a private `ChainSource` + `TxTracker`, sets up a
single-hash subscription, fires `onMined` once on
`seen-in-block` reaching the confirmations threshold, fires `onDropped`
on `unseen-for-n-blocks` (the existing tracker event maps to this), and
tears down both source and tracker on terminal state or explicit stop.

**Returns:** an unsubscribe function. Calling it before terminal cancels
the watch and tears down the internal source/tracker.

### F5 — `replaceTransaction` top-level export

Provex upstream item 5 (`upstream-candidates.md:50-51`).
Placement reasoning: see project memory
`transaction-verbs-package-placement.md`.

**Shape: caller-provides-newGas. Lives in `tx-tracker`.**

```ts
import { replaceTransaction } from '@valve-tech/tx-tracker'

interface ReplaceTransactionOptions {
  /**
   * The original transaction request — must include `nonce` and the
   * full calldata that the caller submitted previously. Caller is
   * responsible for caching this at submit time.
   */
  original: {
    to: Address
    nonce: number
    data?: Hex
    value?: bigint
    chainId?: number
  }
  walletClient: WalletClient
  /**
   * Replacement gas params. Caller computes the bump (via
   * `@valve-tech/gas-oracle`'s helper, viem's, or their own math).
   * `tx-tracker` does NOT compute the EIP-1559 ≥10% bump rule —
   * that helper ships separately.
   */
  newGas: {
    maxFeePerGas: bigint
    maxPriorityFeePerGas: bigint
  }
}

function replaceTransaction(
  options: ReplaceTransactionOptions
): Promise<Hex>  // returns the new tx hash
```

**Hard constraint:** tx-tracker MUST NOT import from `@valve-tech/gas-oracle`.
Sibling packages, kept independent. The bump-rule helper lands in
gas-oracle in a future release as part of `upstream-candidates.md` items
1–3.

### F6 — WS-push transports in chain-source

Spec reference: `docs/tx-tracker-spec.md` §7 (capability-disclosure
contract, already in place).

**Note: capability fields already exist.** `Capabilities.newHeads` and
`Capabilities.newPendingTransactions` already discriminate
`'subscription' | 'poll-only' | 'unavailable'` and the probe in
`packages/chain-source/src/capabilities.ts` already detects WS
structurally via `typeof transport.subscribe === 'function'`. The work
in F6 is **wiring the actual subscribe paths** that the disclosure
already promises — today the source returns `'subscription'` from the
probe but always falls back to interval polling regardless (see comment
in `source.ts:30-35`).

The implementation:

1. In `subscribeBlocks`, when `capabilities().newHeads === 'subscription'`
   and there's no live block-subscribe yet, open one via
   `client.transport.subscribe({ method: 'eth_subscribe', params: ['newHeads'] }, …)`.
   Pipe the head events into `blockSubs.emit(...)` after fetching the
   full block (`source.getBlock(blockNumber)`) — the existing dedup-by-
   hash machinery in `source.ts` makes the push and poll paths coexist
   safely.
2. In the mempool path, do the same for `'newPendingTransactions'`,
   but normalize to **hash-only**: the WS notification carries a hash,
   tx-tracker (the only consumer that cares per-hash) calls
   `source.getTransaction(hash)` if it needs full data.
3. On subscribe failure (`transport.subscribe` throws or returns null),
   downgrade the cached capability to `'poll-only'` and emit a one-shot
   warning via `onError`. The poll cycle continues unchanged.

**Event-side impact:** tx-tracker emits `source: 'subscription'` on
events that came from the WS path. The capability-matrix matrix tests
already assert this contract — they currently exercise an "all-caps"
profile that's effectively a no-op on the source field; this release
makes that profile real by routing through actual WS subscription.

**Reorg behavior is unchanged.** Hash-based dedup at the source still
applies — push and poll can race, the source picks the first
observation per hash and ignores the second.

## 7. Risks and unknowns

- **WS provider variance.** `eth_subscribe('newPendingTransactions')`
  returns hash-only on most providers, full-tx on a few (Infura's
  `alchemy_pendingTransactions` is non-standard). The capability probe
  needs to detect hash-only and surface that distinctly, OR we
  normalize to hash-only and let consumers refetch via
  `source.getTransaction(hash)`. **Decision: normalize to hash-only**;
  if there's consumer demand for full-tx push later, surface as a
  separate capability `subscribeNewPendingFullTx`.
- **F3 `group-failed` is one-shot.** A group with 3 members where
  members 1 and 2 both fail emits `group-failed` only for member 1.
  This is intentional (group is terminal after first failure) but
  worth calling out — consumers who want to see all failures should
  use per-hash subscribe alongside the group.
- **F5 has no protection against malformed `original`.** If the caller
  passes a bad nonce, viem will surface the wallet error as-is. The
  helper does not validate beyond constructing the request.
